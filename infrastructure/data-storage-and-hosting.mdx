---

Superadmission processes three categories of sensitive data simultaneously: academic identity linked to government-issued credentials, financial transactions, and allocation decisions that affect millions of students. How that data is stored — where, for how long, and under what access rules — is not a secondary concern. It is a design decision made at the architecture level.

---

## Where data is stored

All student data is stored within India. The hosting infrastructure is NIC Cloud (National Informatics Centre) and equivalent Indian cloud providers. There is no foreign cloud dependency for any data classified as sensitive under the Digital Personal Data Protection Act, 2023.

This is a hard constraint, not a preference. A system processing Aadhaar-linked identity, DigiLocker-fetched government documents, and UPI payment references for tens of millions of Indian students has no business routing that data through foreign infrastructure. Continuity of the system cannot depend on a foreign provider's availability or policy decisions.

---

## Data categories and retention

| Category | What it includes | Retention period |
|---|---|---|
| Identity record | APAAR ID, name, DOB, gender, address (from Aadhaar) | Duration of active profile + 3 years |
| Academic documents | DigiLocker-fetched marksheets, certificates | Deleted 90 days post-admission session |
| Structured academic data | Marks, subjects, board, year (extracted from documents) | Duration of active profile + 3 years |
| Exam scores | Rank, percentile, score (from NTA/state body APIs) | Duration of active profile + 3 years |
| Payment records | UPI reference, amount, status, timestamp | 7 years (financial record requirement) |
| Allocation records | Preference list, allocation output, decision trail | 7 years (audit and grievance requirement) |
| Session logs | Login events, action timestamps, IP address | 90 days rolling |
| Aadhaar number | Not stored at any point | — |
| Payment credentials | Card numbers, bank details, UPI IDs — not stored | — |

Documents are the most sensitive category and carry the shortest retention. Once an admission session closes and 90 days have passed, the DigiLocker-fetched document files are deleted. The structured data extracted from them — marks, dates, category — is retained as part of the student's academic record, but the document itself is not held longer than necessary.

## Data partitioning

Student data is partitioned by counselling cycle. A student's profile for the 2025 admission cycle is stored separately from any future cycle. Counselling authorities access data only for their own counselling. An institution connected to one state counselling cannot access data from a different authority's counselling.

Access is role-based and scoped:

| Role | Access scope |
|---|---|
| Student | Own profile only |
| Institution admin | Applications to their institution, within their counselling |
| Counselling authority | All students registered in their counselling |
| Pravesh AI (system) | Read access for verification and recommendation; no write access to identity fields |
| System administrator | Infrastructure access; no access to student content data |

No role has broad access to the full student database. System administrators can access infrastructure logs but not student identity or academic records. This separation is enforced at the database layer, not just at the application layer.

---

## Backup and recovery

Data is replicated across at least two geographically separate Indian data centres. Recovery point objective (RPO) — the maximum data loss acceptable in a failure — is four hours. Recovery time objective (RTO) — the maximum acceptable downtime — is two hours for critical functions (allocation, payment confirmation, identity verification).

Backup integrity is verified on a weekly automated schedule. Restoration is tested quarterly.

---

## What is explicitly not stored

- Aadhaar numbers (only APAAR hash is retained)
- Biometric data of any kind
- UPI IDs, bank account details, card numbers
- Counselling authority internal communications
- Any data from DigiLocker that is not required for the admission workflow

The principle is: if it is not needed for the admission process, it is not collected. If it has been collected and is no longer needed, it is deleted on schedule.
