## Current Journey

```mermaid
flowchart TD
    R[Exam result received] --> A1[Register on JoSAA portal\nEnter all personal details\nPay registration fee]
    R --> A2[Register on State CET portal\nRe-enter all personal details\nPay registration fee again]

    A1 --> B1[Upload documents to JoSAA\nPhotograph, mark sheets,\ncategory certificate, domicile]
    A2 --> B2[Upload same documents to State CET\nDifferent size limits\nDifferent file format requirements]

    B1 --> C1[JoSAA: Choice filling\nDeadline: Day 5, 11:59 PM]
    B2 --> C2[State CET: Choice filling\nDeadline: Day 7, 5:00 PM]

    C1 --> D1[JoSAA Round 1 allotment\nCheck portal independently]
    C2 --> D2[State CET Round 1 allotment\nCheck portal independently]

    D1 --> E{Compare allotments\nManually}
    D2 --> E

    E --> F1[Accept better offer\nPay acceptance fee on that portal]
    F1 --> F2[Withdraw from other system\nLocate withdrawal option\nMiss deadline risk]
    
    F2 --> G[Track reporting deadlines\nSeparately per system]
    G --> H[Report to institution\nCarry originals\nVerification again]
```

**What the student manages manually in the current journey:**

| Task | Times performed |
| --- | --- |
| Account creation | Once per counselling system |
| Personal information entry | Once per counselling system |
| Document upload | Once per counselling system |
| Deadline tracking | Independently per system, per round |
| Portal login to check status | Daily, across multiple portals |
| Allotment comparison | Manually, with no aggregated view |
| Withdrawal from non-accepted systems | Manually, with risk of missing deadline |
| Choice Filling | Without any structured guidance |

## Intended Journey

```mermaid
flowchart LR
    R[Exam result received] --> A[Create Superadmission profile\nOnce\nPersonal details entered once]
    
    A --> B[Upload documents once\nVerified centrally\nStatus available to all systems]
    
    B --> C[Eligibility surfaced automatically\nJoSAA eligible · State CET eligible\nNo manual eligibility research]
    
    C --> D[Choice filling — unified interface\nBrowse JoSAA and State CET options\nIn one place, same interface]
    
    D --> E[Deadlines displayed together\nPriority sorted by urgency\nNotifications sent proactively]
    
    E --> F[Allotments received\nBoth displayed in one dashboard\nComparison view available]
    
    F --> G[Accept preferred offer\nWithdrawal from other system\nGuided, one click]
    
    G --> H[Reporting deadline tracked\nDocuments pre-verified\nInstitution notified]
    
    H --> I[Admission confirmed]
```

## Key Differences

<CardGroup cols={2}>
  <Card title="One Profile" icon="user-check">
    The student creates one identity-linked profile. Personal details, academic record, category status entered once, referenced everywhere. No re-entry across systems.
  </Card>

  <Card title="Documents Verified Once" icon="file-check">
    Documents are uploaded and verified once. The verified status is referenced by all counselling systems the student participates in. No re-upload, no re-verification.
  </Card>

  <Card title="Unified Deadline View" icon="calendar-check">
    All active deadlines, choice fill, allotment response, acceptance, reporting across all counselling systems in a single view, sorted by urgency.
  </Card>

  <Card title="Guided Decision Points" icon="compass">
    At each major decision  the guidance layer surfaces relevant information. The student makes the decision. The system ensures they have the context to make it well.
  </Card>
</CardGroup>

## Student Control and Data

The proposed architecture is designed around the principle that the student owns their profile and their data.

**What the student controls:**

- Which counselling systems to connect to their profile
- Which documents to share with which systems
- Choice filling preferences - the system assists but does not pre-fill or lock preferences
- Acceptance decisions - no decision is made automatically on the student's behalf
